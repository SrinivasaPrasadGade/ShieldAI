## Secrets Management Changes & Fixes

Branch: `fix/secure-firebase-credentials` (merged into `main`)

1. docker-compose.yml:
   - Removed the hardcoded volume mounts for `./backend/firebase-credentials.json` from the `api`, `worker`, and `realtime` services. This prevents physical secret files from being required and mounted on host machines.
   - Replaced `FIREBASE_CREDENTIALS_PATH` environment variables with `FIREBASE_CREDENTIALS_JSON` and `FIREBASE_CREDENTIALS_B64` across all services to allow securely injecting credentials via the environment.

2. backend/config.py:
   - Added `FIREBASE_CREDENTIALS_JSON` and `FIREBASE_CREDENTIALS_B64` to the `Settings` class so `pydantic` can safely load these values from the environment or `.env` files.

3. backend/models/database.py:
   - Updated the `get_firestore_client()` initialization function to prioritize checking for the `FIREBASE_CREDENTIALS_B64` or `FIREBASE_CREDENTIALS_JSON` environment variables.
   - Added parsing logic using the `json` and `base64` modules. This converts the environment string into a Python dictionary, which is then loaded into `credentials.Certificate(cred_dict)`.
   - Maintained fallback support for reading the physical `firebase-credentials.json` file (useful for local development convenience if a developer hasn't set up the environment variables).
   - Preserved the final fallback to Application Default Credentials (ADC) for Google Cloud Run and GKE zero-key workload identity environments.

4. .gitignore:
   - Added `*.env` and `.env.*` to completely block all environment configuration files from being tracked by git.
   - Added a global `firebase-credentials.json` ignore rule to prevent accidental commits of this file regardless of which directory it resides in.

## Authentication & Authorization Security Updates

Branch: `fix/jwt-auth-rbac` (merged into `main`)

1. backend/middleware.py:
   - Removed the insecure, hardcoded `demo-valid-token` API key validation logic from `AuthMiddleware`.
   - Implemented real JWT verification using `firebase_admin.auth.verify_id_token(token)`.
   - Added Role-Based Access Control (RBAC). The middleware now verifies that the decoded JWT contains a `role` claim of `law_enforcement` or an `admin` claim set to `true` before granting access to protected routes like `/api/graph` and `/api/geo`. It returns `401 Unauthorized` for invalid tokens and `403 Forbidden` for insufficient permissions.

2. backend/models/database.py & backend/main.py:
   - Refactored the Firebase app initialization into a new, dedicated `init_firebase()` function in `database.py`.
   - Added an explicit call to `init_firebase()` in the FastAPI `lifespan` handler within `main.py` to ensure the Firebase Admin SDK is fully initialized at startup and ready to verify tokens on the very first incoming request.

3. frontend/src/App.jsx:
   - Removed the mocked token insertion `localStorage.setItem('auth_token', 'demo-valid-token')` when attempting to access the law enforcement dashboard without credentials.
   - The application now strictly enforces authentication for the dashboard, alerting the user and blocking client-side routing if a valid Firebase JWT `auth_token` is not present in local storage.

## Currency Image Validation ŌĆö Firestore Persistence Fix

Branch: `fix/currency-firestore-persistence` (merged into `main`)

1. backend/services/currency_analyzer.py:
   - Added `_CITY_COORDS` module-level lookup table mapping 11 major Indian cities to their (lat, lng) coordinates.
   - Added `_parse_location(location)` helper function that normalises the optional `location` string passed to the Celery worker into a `{lat, lng, city}` dict matching the Firestore document schema expected by `/api/currency/ficn-map`. Handles four input shapes: `None`, a plain city name (e.g. `"Mumbai"`), a `"lat,lng"` pair, and a `"city,lat,lng"` triple.
   - Added `_persist_to_firestore()` async method on `CurrencyAnalyzer`. After every successful verification it writes a complete document to the Firestore `currency_checks` collection (fields: `id`, `denomination`, `verdict`, `confidence`, `failed_features`, `gemini_analysis`, `image_url`, `location`, `submitted_by`, `created_at`). The write is best-effort ŌĆö a failure is logged via `structlog` but does not alter the task result stored in SQLite.
   - Updated `run_verification()` to call `_persist_to_firestore()` as step 5, after `task_store.update_task(task_id, status="complete", ...)` succeeds. This closes the gap where `/api/currency/stats` and `/api/currency/ficn-map` only reflected seeded demo data instead of real uploads.

2. docs/CODEBASE_AUDIT.md:
   - Updated the Counterfeit Currency Verification limitations section to mark the Firestore persistence gap as resolved and reference the fix branch `fix/currency-firestore-persistence`.
